Vulkan Renderer — Geometry Rendering

Building on the working frame loop (clear color + present), this plan adds everything needed to draw colored triangles and lines — rectangles, circles, polygons, all of drawShape. The approach mirrors the Metal renderer's batch-accumulate-flush pattern, adapted to Vulkan pipelines and host-visible vertex buffers. Textures are out of scope for this pass.

Phases

  1. Already Done  — frame loop, clear color, present
  2. Phase 1 — Vertex Types & Batch  vertex.zig, geometry_batch.zig
  3. Phase 2 — Graphics Pipeline  support/Pipeline.zig

  5. Phase 4 — Wire Up Renderer  vulkan_renderer.zig
  6. Phase 5 — Shader Build Step  build.zig
Design goal: Match the Metal renderer's API exactly — drawShape accumulates into a CPU-side batch, endFrame uploads and flushes it. Nothing about the engine-level call sites changes. Only the backend plumbing differs.

Data Flow

Per-frame lifecycle: beginFrame() │ batch.clear() ← discard last frame's geometry │ vkWaitForFences / vkAcquireNextImage │ vkBeginCommandBuffer └── vkCmdBeginRenderPass (clears color) ↓ game code calls renderer.drawGeometry(...) drawShape(shape, transform, fill, stroke, ctx) └── batch.addShape(...) ← tessellate + clip → Vertex slice endFrame() │ flushBatch() │ vertex_buffer.upload(batch.vertices) ← memcpy to mapped GPU mem │ vkCmdBindVertexBuffers(vertex_buffer) │ for each draw_call in batch.draw_calls: │ if topology changed → vkCmdBindPipeline (triangle or line) │ vkCmdDraw(vertex_count, 1, vertex_start, 0) │ vkCmdEndRenderPass │ vkEndCommandBuffer └── vkQueueSubmit → vkQueuePresentKHR

Vertex Format

Identical to the Metal renderer's Vertex struct — the batch code is shared logic.

Field Type Offset VkFormat Location
position [2]f32 0 VK_FORMAT_R32G32_SFLOAT 0
color [4]f32 8 VK_FORMAT_R32G32B32A32_SFLOAT 1

Stride: @sizeOf(Vertex) = 24 bytes. Use extern struct to guarantee no padding.

Phase 1

Vertex Types & Geometry Batch new

Two new files. vertex.zig defines the shared data types used by both the batch and the pipeline. geometry_batch.zig is a direct port of Metal's batch accumulator — same tessellation logic, same draw call structure, different primitive type enum.

vertex.zig

  • Vertex = extern struct { position: [2]f32, color: [4]f32 }
  • Topology = enum { triangle, line }
Why a separate file? Both Pipeline.zig (in support/) and geometry_batch.zig need the Vertex layout — Pipeline needs it to describe binding/attribute descriptions, batch needs it to build vertex slices. Putting it in a neutral file avoids any circular import risk between those two.

geometry_batch.zig — struct

  • DrawCall = struct { topology: Topology, vertex_start: u32, vertex_count: u32 }
  • GeometryBatch = struct { vertices: std.ArrayList(Vertex), draw_calls: std.ArrayList(DrawCall), gpa: Allocator }
  • init(gpa) GeometryBatch — .vertices = .empty, .draw_calls = .empty
  • deinit(self) — deinit both lists
  • clear(self) — clearRetainingCapacity on both lists

geometry_batch.zig — addShape dispatch

Port Metal's geometry_batch.zig verbatim, replacing MTLPrimitiveType with Topology and primitive_type with topology in DrawCall. Everything else — the inline switch dispatch, makeVertex, makeScreenVertex, and all shape handlers — is identical.

  • addShape — delegates to addShapeDispatch
  • addShapeDispatch — inline switch on ShapeData tag, calls addLine, addTriangle, addRectangle, addCircle, addPolygon, addEllipse
  • makeVertex(point, xform, ctx, color) Vertex — transformPoint + worldToClipSpace
  • makeScreenVertex(point, xform, ctx, color) Vertex — offset transform + screenToClipSpace
  • addLine — 2 vertices, topology: .line
  • addTriangle — 3 fill vertices (.triangle) + 6 outline vertices (.line)
  • addRectangle — 6 fill vertices (2 tris) + 8 outline vertices (4 lines)
  • addCircle — 32 segments, 3 fill vertices per segment, 2 outline vertices per segment
  • addPolygon — uses triangle_cache for fill, iterates edges for outline
  • addEllipse — stub (same as Metal, not yet implemented)
Imports for geometry_batch.zig: @import("vertex.zig") for Vertex/Topology, @import("../../renderer.zig") for Color/Transform/ShapeData, @import("math") for WorldPoint/ScreenPoint (module import, injected by build), @import("../../RenderContext.zig"), @import("../../geometry_utils.zig") for transform/clip helpers.
Phase 2

Graphics Pipeline new

Pipeline.zig wraps a VkPipeline + VkPipelineLayout. One pipeline per topology — triangles use VK_PRIMITIVE_TOPOLOGY_TRIANGLE_LIST, lines use VK_PRIMITIVE_TOPOLOGY_LINE_LIST. Viewport and scissor are baked in (not dynamic) — resize recreates the pipelines alongside the swapchain.

Pipeline.zig — struct fields

  • handle: *vk.struct_VkPipeline_T
  • layout: *vk.struct_VkPipelineLayout_T

init(dev, render_pass, vert_spv, frag_spv, topology, extent) !Self

Shader modules

  • createShaderModule(dev, code: []const u8) !*VkShaderModule — private helper
  • VkShaderModuleCreateInfo: codeSize = code.len, pCode = @ptrCast(@alignCast(code.ptr))
  • Create vert and frag modules; defer destroy both before returning (modules consumed by pipeline creation)
  • Two VkPipelineShaderStageCreateInfo entries: VK_SHADER_STAGE_VERTEX_BIT + VK_SHADER_STAGE_FRAGMENT_BIT, both with pName = "main"

Vertex input state

  • One VkVertexInputBindingDescription:
    • binding = 0
    • stride = @sizeOf(Vertex) (24 bytes)
    • inputRate = VK_VERTEX_INPUT_RATE_VERTEX
  • Two VkVertexInputAttributeDescription:
    • location=0: format = VK_FORMAT_R32G32_SFLOAT, offset = @offsetOf(Vertex, "position")
    • location=1: format = VK_FORMAT_R32G32B32A32_SFLOAT, offset = @offsetOf(Vertex, "color")

Input assembly

  • topology = switch (topology) { .triangle => VK_PRIMITIVE_TOPOLOGY_TRIANGLE_LIST, .line => VK_PRIMITIVE_TOPOLOGY_LINE_LIST }
  • primitiveRestartEnable = VK_FALSE

Viewport & scissor state (baked)

  • VkViewport: x=0, y=0, width/height from extent, minDepth=0.0, maxDepth=1.0
  • VkRect2D scissor: offset={0,0}, extent=extent
  • viewportCount = 1, scissorCount = 1

Rasterizer

  • polygonMode = VK_POLYGON_MODE_FILL
  • cullMode = VK_CULL_MODE_NONE — 2D geometry has no consistent winding
  • frontFace = VK_FRONT_FACE_COUNTER_CLOCKWISE
  • lineWidth = 1.0
  • All depth clamp, bias, discard flags: VK_FALSE

Multisample state

  • rasterizationSamples = VK_SAMPLE_COUNT_1_BIT — no MSAA
  • sampleShadingEnable = VK_FALSE

Color blend state (alpha blending)

  • blendEnable = VK_TRUE
  • srcColorBlendFactor = VK_BLEND_FACTOR_SRC_ALPHA
  • dstColorBlendFactor = VK_BLEND_FACTOR_ONE_MINUS_SRC_ALPHA
  • colorBlendOp = VK_BLEND_OP_ADD
  • srcAlphaBlendFactor = VK_BLEND_FACTOR_ONE
  • dstAlphaBlendFactor = VK_BLEND_FACTOR_ZERO
  • alphaBlendOp = VK_BLEND_OP_ADD
  • colorWriteMask = R | G | B | A

Pipeline layout (no descriptors)

  • VkPipelineLayoutCreateInfo with zero descriptor set layouts and zero push constant ranges
  • vkCreatePipelineLayout → self.layout

Pipeline creation

  • pDepthStencilState = null — render pass has no depth attachment
  • pTessellationState = null — no tessellation
  • pDynamicState = null — viewport baked in for now
  • renderPass = render_pass.handle, subpass = 0
  • basePipelineHandle = null — no pipeline derivatives
  • vkCreateGraphicsPipelines(dev, null, 1, &info, null, &pipeline)
  • On failure: destroy pipeline layout before returning error

deinit(dev)

  • vkDestroyPipeline(dev.handle, self.handle, null)
  • vkDestroyPipelineLayout(dev.handle, self.layout, null)
Shader modules can be destroyed immediately after pipeline creation. Vulkan copies the SPIR-V bytecode into the pipeline at creation time. The VkShaderModule handles can and should be destroyed right away — keeping them alive wastes memory and requires extra cleanup. Use defer to destroy them before Pipeline.init returns.
Why two pipelines (triangle + line)? In Vulkan, primitive topology is baked into the pipeline's input assembly state. You cannot switch between triangle and line rendering mid-pipeline without dynamic state extensions (which add driver complexity). Two small pipelines — one per topology — is the standard approach and has zero runtime cost beyond a single vkCmdBindPipeline call when topology changes in the draw call stream.
Phase 3

Vertex Buffer new

A persistently-mapped, host-coherent VkBuffer. Each frame, upload() memcpys the batch's vertex slice into it. No staging buffer, no explicit flush — HOST_COHERENT keeps CPU writes visible to the GPU automatically.

VertexBuffer.zig — struct fields

  • handle: *vk.struct_VkBuffer_T
  • memory: *vk.struct_VkDeviceMemory_T
  • mapped: [*]u8 — persistent CPU pointer from vkMapMemory
  • capacity: usize — max bytes this buffer can hold

init(dev, gpu, size) !Self

  • VkBufferCreateInfo: size = size, usage = VK_BUFFER_USAGE_VERTEX_BUFFER_BIT, sharingMode = VK_SHARING_MODE_EXCLUSIVE
  • vkCreateBuffer → handle
  • vkGetBufferMemoryRequirements → mem_reqs
  • Find a memory type index that satisfies mem_reqs.memoryTypeBits and has both VK_MEMORY_PROPERTY_HOST_VISIBLE_BIT and VK_MEMORY_PROPERTY_HOST_COHERENT_BIT
  • VkMemoryAllocateInfo: allocationSize = mem_reqs.size, memoryTypeIndex from above
  • vkAllocateMemory → memory
  • vkBindBufferMemory(dev, handle, memory, 0)
  • vkMapMemory(dev, memory, 0, VK_WHOLE_SIZE, 0, &ptr) → store as self.mapped
  • On error at any step: clean up already-created resources before returning the error

findMemoryType (private)

  • vkGetPhysicalDeviceMemoryProperties(gpu.handle, &mem_props)
  • Loop 0..mem_props.memoryTypeCount: check bit i set in type_filter AND propertyFlags & required == required
  • Return ?u32 — null if no suitable type found

upload(data: []const u8)

  • const len = @min(data.len, self.capacity)
  • @memcpy(self.mapped[0..len], data[0..len])
  • No flush needed — HOST_COHERENT guarantees GPU visibility

deinit(dev)

  • vkUnmapMemory(dev.handle, self.memory)
  • vkFreeMemory(dev.handle, self.memory, null)
  • vkDestroyBuffer(dev.handle, self.handle, null)
Buffer size: Use 12MB — same as the Metal renderer's vertex buffer. Define as a const in vulkan_renderer.zig: const MAX_VERTEX_BUFFER_SIZE: usize = 1024 * 1024 * 12; The batch already clamps at this limit on the Metal side; the same cap applies here.
Why HOST_COHERENT instead of a staging buffer? A staging buffer would require a separate device-local buffer, a transfer command, and pipeline barriers — significant complexity for geometry that changes every frame. HOST_COHERENT memory is slightly slower on discrete GPUs (no VRAM cache), but for 2D geometry batches it's negligible. Integrated GPUs (common on Linux dev hardware) have unified memory so there's no penalty at all. Optimize with staging if profiling shows it matters.

beginFrame() — add at top

  • self.batch.clear(); — discard last frame before acquiring next image

endFrame() — call flushBatch before vkCmdEndRenderPass

  • Insert self.flushBatch(); after the render pass begins and before vkCmdEndRenderPass

flushBatch() — private

  • Early return if self.batch.draw_calls.items.len == 0
  • Upload vertices: self.vertex_buffer.upload(@as([*]const u8, @ptrCast(vertices.ptr))[0..byte_len])
  • Bind vertex buffer: vkCmdBindVertexBuffers(cmd, 0, 1, &buf_handle, &offset_zero)
    • var buf_handle: vk.VkBuffer = self.vertex_buffer.handle (coerce *T → ?*T)
    • const offset_zero: vk.VkDeviceSize = 0
  • Iterate draw calls, binding pipeline only when topology changes:
    • Track var current_topology: ?Topology = null
    • If call.topology != current_topology: vkCmdBindPipeline(cmd, VK_PIPELINE_BIND_POINT_GRAPHICS, pipeline.handle)
    • vkCmdDraw(cmd, call.vertex_count, 1, call.vertex_start, 0)

drawShape() — replace no-op stub

  • self.batch.addShape(shape, transform, fill_color, stroke_color, stroke_width, ctx) catch |err| log.err(...)

Update types.zig

  • pub const Pipeline = @import("support/Pipeline.zig");
  • pub const VertexBuffer = @import("support/VertexBuffer.zig");
  • const vtx = @import("vertex.zig"); pub const Vertex = vtx.Vertex; pub const Topology = vtx.Topology;
flushBatch goes inside the render pass. vkCmdDraw is only valid while a render pass is active. The correct order in endFrame is: vkCmdBeginRenderPass (in beginFrame) → flushBatch() → vkCmdEndRenderPass. Calling draw commands after vkCmdEndRenderPass is a validation error.
vkCmdBindVertexBuffers needs a VkBuffer (nullable pointer), not the unwrapped handle. self.vertex_buffer.handle is *vk.struct_VkBuffer_T. Zig will not implicitly coerce it to vk.VkBuffer (?*struct_VkBuffer_T) when taking the address of it. Store it in a local variable first: var buf: vk.VkBuffer = self.vertex_buffer.handle; then pass &buf.
Phase 5

Shader Build Step modify

glslc is available at /usr/bin/glslc. The build step compiles both GLSL shaders to SPIR-V at build time, co-locates them with a generated Zig wrapper, and injects the wrapper as an anonymous import (vulkan_shaders) into the renderer module.

Shaders to create

  • src/renderer/gpu/vulkan/shaders/geometry.vert 
    #version 450
layout(location = 0) in vec2 inPosition;
layout(location = 1) in vec4 inColor;
layout(location = 0) out vec4 fragColor;
void main() {
    gl_Position = vec4(inPosition, 0.0, 1.0);
    fragColor = inColor;
}
  • src/renderer/gpu/vulkan/shaders/geometry.frag 
    #version 450
layout(location = 0) in vec4 fragColor;
layout(location = 0) out vec4 outColor;
void main() { outColor = fragColor; }

build.zig — add compileVulkanShaders function

  • Add at the bottom of build.zig: 
    fn compileVulkanShaders(
    b: *std.Build,
    renderer_module: *std.Build.Module,
    target: std.Build.ResolvedTarget,
    optimize: std.builtin.OptimizeMode,
) void {
    const vert = b.addSystemCommand(&.{"glslc"});
    vert.addFileArg(b.path("src/renderer/gpu/vulkan/shaders/geometry.vert"));
    vert.addArg("-o");
    const vert_spv = vert.addOutputFileArg("geometry_vert.spv");

    const frag = b.addSystemCommand(&.{"glslc"});
    frag.addFileArg(b.path("src/renderer/gpu/vulkan/shaders/geometry.frag"));
    frag.addArg("-o");
    const frag_spv = frag.addOutputFileArg("geometry_frag.spv");

    const wf = b.addWriteFiles();
    _ = wf.addCopyFile(vert_spv, "geometry_vert.spv");
    _ = wf.addCopyFile(frag_spv, "geometry_frag.spv");
    const shaders_zig = wf.add("shaders.zig",
        \\pub const geometry_vert = @embedFile("geometry_vert.spv");
        \\pub const geometry_frag = @embedFile("geometry_frag.spv");
    );

    renderer_module.addAnonymousImport("vulkan_shaders", .{
        .root_source_file = shaders_zig,
        .target = target,
        .optimize = optimize,
    });
}
  • Call it from build() after configurePlatformModule: 
    if (target.result.os.tag == .linux and selected_renderer == .vulkan) {
    compileVulkanShaders(b, modules[@intFromEnum(M.renderer)], target, optimize);
}
How the anonymous import works: addWriteFiles creates a temporary build-cache directory. wf.add("shaders.zig", ...) writes the Zig wrapper into that directory. wf.addCopyFile(spv, "geometry_vert.spv") copies the compiled SPIR-V into the same directory. When Zig compiles shaders.zig, its @embedFile calls resolve relative to that directory — finding the copied .spv files. The result is accessible in vulkan_renderer.zig as @import("vulkan_shaders").geometry_vert (a []const u8).
Only runs on Linux Vulkan builds. The if guard prevents the glslc system command from running on macOS (Metal) builds where glslc may not be installed.

Verification

In order:

  • zig build zixelart compiles cleanly — no errors in new files
  • Both .spv files are generated in the build cache (no glslc errors)
  • Window opens and displays the clear color — existing behavior preserved
  • A scene or tool that calls drawGeometry now renders visible geometry (rectangles, circles, etc.)
  • Lines render correctly (outlines on rectangles, polygon edges)
  • Alpha blending works — overlapping semi-transparent shapes composite correctly
  • No Vulkan validation errors in the draw path
  • Resize still works — the pipelines are swapchain-extent-dependent, so resize() will need to recreate them (see Out of Scope below)

Out of Scope (this pass)

  • Textures — drawTextureQuad stays a stub; requires descriptor sets, samplers, image layout transitions, and a texture pipeline
  • Dynamic viewport — viewport and scissor are baked into each pipeline; resize requires pipeline recreation in addition to swapchain recreation
  • Depth buffer — no depth testing for 2D rendering; add later if 3D or z-order sorting is needed
  • Index buffers — not needed for the current tessellation output; a future optimization
  • Multiple draw batches per frame — single upload per frame; sufficient for now
  • Staging buffer upload — host-coherent mapping is adequate; optimize to staging + device-local if GPU profiling shows it's a bottleneck
What comes after: With geometry working, the natural next step is textures — drawTextureQuad needs a second pipeline with UV coordinates, a descriptor set layout binding a combined image sampler, and a staging upload path for texture data. The batch infrastructure from this pass remains intact.
