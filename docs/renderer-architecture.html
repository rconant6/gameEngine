Overview

The renderer is a multi-backend abstraction layer. Currently only the Metal backend is active (macOS). Vulkan and OpenGL are stubs. A legacy CPU software renderer exists but is disabled.

Frame Pipeline

Every frame follows this exact sequence. The engine orchestrates; systems submit draw calls; the backend batches and flushes to the GPU.

  1. renderer.beginFrame() — clear geometry batch, get next CAMetalDrawable, create MTLCommandBuffer
  2. RenderSys queries Transform + Sprite → calls drawGeometry() for each visible entity in world space
  3. RenderSys queries Transform + Text → calls drawText() for each text entity
  4. RenderSys queries UIElement + Sprite → calls drawGeometry() with screen-space anchoring
  5. DebugDraw submits overlay geometry (collision bounds, velocity arrows, camera frustums)
  6. renderer.endFrame() → flushBatch(): copy vertices to GPU buffer, create render pass, issue all draw calls, present drawable, commit command buffer

Batch Flush Detail

  1. Copy GeometryBatch.vertices into shared MTLBuffer (12MB max)
  2. Create MTLRenderPassDescriptor with color attachment (clear color, load: clear, store: store)
  3. Create MTLRenderCommandEncoder from command buffer + render pass
  4. Set geometry pipeline state on encoder
  5. Bind vertex buffer at index 0
  6. For each DrawCall: issue drawPrimitives(type, start, count)
  7. End encoding, present drawable, commit command buffer

Coordinate Spaces

Three coordinate systems with conversion utilities. src/renderer/geometry_utils.zig

World Space Game coordinates, infinite range, floats. Camera-relative. Y+ is up.
Screen Space Pixel coordinates, (0,0) at top-left, (w,h) at bottom-right. Y+ is down. Used for UI.
Clip Space (NDC) GPU normalized device coordinates, (-1,-1) to (+1,+1). Center is origin. Automatic — game code never uses this directly.

Text Rendering

Converts font glyphs to triangles and renders them as geometry. No texture atlases — pure vector rendering. src/renderer/text.zig

text.zig
drawText(renderer, font, text, position, scale, color, ctx) // world space
drawTextScreen(renderer, font, text, position, scale, color, ctx) // screen space
drawGlyph(renderer, font, glyph_index, glyph, scale, pos, color, ctx) // single glyph

Text Rendering Flow

  1. For each character in the text string, look up ASCII value in Font.char_to_glyph map
  2. Retrieve FilteredGlyph shape from Font.glyph_shapes
  3. Lazy-build triangles from glyph outlines (cached in Font.glyph_triangles after first use)
  4. For each triangle: apply position, scale, and transform → call drawGeometry()
  5. Advance X cursor by glyph's advance_width (scaled to render size)

Y-flip note: Text rendering flips Y (y = -y * scale + y_pos) because glyph coordinates use math convention (Y+ up) while screen coordinates use graphics convention (Y+ down).

Triangulation

Converts arbitrary polygons to triangles for GPU rendering. Two algorithms: monotone partition (general polygons) and ear clipping (font glyphs).

Monotone Polygon Partition (src/renderer/triangulation.zig)

triangulate()
triangulate(allocator, points) → [][3]Point
Returns counter-clockwise oriented triangles
Handles degenerate cases (< 3 vertices)

Algorithm steps:

  1. Classify each vertex as Start, End, Split, Merge, or Regular
  2. Sort vertices by Y coordinate (sweep-line)
  3. Maintain active edge set, generate diagonals to partition into monotone pieces
  4. Triangulate each monotone piece with a stack-based algorithm

Ear Clipping (src/assets/font/ear_clipping.zig)

EarClipper
triangulate(contours) → []Triangle
Handles outer contours + holes (bridges holes via rightmost vertex)
Classifies vertices as convex/reflex, clips ear tips iteratively
Used for font glyph triangulation (complex multi-contour shapes)
Polygon (uses triangulation) Text (glyph triangulation)

Initialization Flow

  1. Create MTLDevice (system default GPU)
  2. Create MTLCommandQueue from device
  3. Get CAMetalLayer from platform window handle
  4. Load compiled .metallib shader library from disk
  5. Create geometry pipeline: vertex_main + fragment_main
  6. Create texture pipeline: texture_vertex_main + texture_fragment_main (with alpha blending)
  7. Allocate 12MB shared MTLBuffer for vertex data

Shaders

Two render pipelines defined in src/platform/macos/swift/shaders.metal. Compiled to .metallib at build time.

Geometry Pipeline

vertex_main + fragment_main
Input (VertexIn) position: float2 [[attribute(0)]] // clip space
color: float4 [[attribute(1)]] // RGBA
Vertex Shader position → float4(position, 0.0, 1.0) // Z=0 (ortho), W=1
color → pass through
Fragment Shader return input.color // no textures, no lighting

Texture Pipeline

texture_vertex_main + texture_fragment_main
Input (TextureVertexIn) position: float2 [[attribute(0)]]
texcoord: float2 [[attribute(1)]]
Vertex Shader position → float4(position, 0.0, 1.0)
texcoord → pass through
Fragment Shader constexpr sampler(mag_filter::nearest, min_filter::nearest)
return tex.sample(sampler, texcoord) // nearest-neighbor (pixel-perfect)
Blending (set in pipeline descriptor) dst = src * srcAlpha + dst * (1 - srcAlpha) // standard alpha blend

Texture Vertex Descriptor Attribute 0: float2 at offset 0   Attribute 1: float2 at offset 8   Stride: 16 bytes

Vertex Budget

Understanding the limits of the current batching system.

Buffer Size MAX_VERT_SIZE = 1024 × 1024 × 12 = 12,582,912 bytes (12MB)
Geometry Vertex 24 bytes per vertex (8 position + 16 color)
Max vertices per frame: ~524,288
Texture Vertex 16 bytes per vertex (8 position + 8 texcoord)
Max textured vertices per frame: ~786,432
Practical Limits (geometry only) Circles (32 segments, fill+outline): ~160 verts → ~3,276 circles/frame
Rectangles (fill+outline): ~14 verts → ~37,449 rects/frame
Triangles (fill+outline): ~9 verts → ~58,254 triangles/frame

Key Design Notes

No depth testing — painter's algorithm (draw order = submission order)
No instancing — each shape gets unique vertices
No compute shaders — all geometry built on CPU
All triangles are counter-clockwise (CCW front-facing)
Circle segments fixed at 32 (TODO: adaptive based on screen size)
Single render pass per frame (geometry), second pass for textures (when used)
