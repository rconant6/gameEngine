🎮 Engine API Layers & Packaged Demos Philosophy

The Vision

Shitty versions right now at all levels, but legit overtime.

Build an engine with progressive API layers that serve everyone from complete beginners to senior Zig engineers who want full control. Each layer gets better and better over time. Ship the engine with packaged demo games and exposed source code - a cool thing to learn how to do!

🎯 Core Philosophy

Progressive Disclosure of Complexity

Don't force beginners to understand the entire engine. Don't block experts from accessing low-level details. Provide layers that meet users where they are.

Learn By Example

Ship real, working games alongside the engine. Expose the source. Let developers run them, play with them, break them, and learn from them.

Iterative Excellence

Start with working implementations at each layer. Improve them over time. Version 1 doesn't need to be perfect - it needs to be functional and teachable.

📚 API Layer Progression

Layer 1: Absolute Beginner ("Just Make It Work")

Target User: New to Zig, new to game dev, just wants to see something on screen

const engine = @import("game_engine"); pub fn main() !void { var game = try engine.init("My First Game"); defer game.deinit(); try game.run(); }

What This Hides:

  • Memory allocation strategies
  • Render loop details
  • ECS setup and registration
  • Window creation and event polling
  • Resource loading

What This Provides:

  • Default game loop
  • Basic rendering
  • Window with reasonable defaults
  • Automatic cleanup
Layer 2: Guided Control ("Configure The Basics")

Target User: Comfortable with Zig basics, wants to customize window, configure settings

const engine = @import("game_engine"); pub fn main() !void { var game = try engine.init(.{ .title = "My Platformer", .width = 1280, .height = 720, .target_fps = 60, .allocator = std.heap.page_allocator, }); defer game.deinit(); try game.addSystem(myUpdateSystem); try game.addSystem(myRenderSystem); try game.run(); }

What This Exposes:

  • Configuration options
  • Custom system registration
  • Allocator choice
  • Basic customization hooks
Layer 3: Loop Control ("I'll Handle The Loop")

Target User: Wants control over game loop timing, update/render separation

const engine = @import("game_engine"); pub fn main() !void { var game = try engine.create(.{ /* config */ }); defer game.deinit(); try game.setup(); while (game.isRunning()) { try game.pollEvents(); try game.update(game.getDeltaTime()); try game.render(); } }

What This Exposes:

  • Manual game loop control
  • Event polling
  • Delta time access
  • Separate update/render calls
Layer 4: Full Engine Control ("I Know What I'm Doing")

Target User: Senior Zig dev, wants full control of memory, systems, threading

const engine = @import("game_engine"); const std = @import("std"); pub fn main() !void { var gpa = std.heap.GeneralPurposeAllocator(.{}){}; defer _ = gpa.deinit(); var window = try engine.Window.create(gpa.allocator(), .{ .title = "Custom Game", .width = 1920, .height = 1080, }); defer window.destroy(); var world = try engine.ECS.World.init(gpa.allocator()); defer world.deinit(); var renderer = try engine.Renderer.init(gpa.allocator(), window); defer renderer.deinit(); // Full manual control of everything while (window.isOpen()) { window.pollEvents(); // Custom delta time calculation const dt = myCustomTimingLogic(); // Manual system execution try myPhysicsSystem(&world, dt); try myGameLogicSystem(&world, dt); renderer.beginFrame(); try myRenderSystem(&world, &renderer); renderer.endFrame(); } }

What This Exposes:

  • Direct component access
  • Manual memory management
  • Custom render pipelines
  • Threading control
  • Everything - no training wheels

📦 Demo Game Philosophy

Source Code Exposure
  • All demo games live in examples/ directory
  • Fully documented source code
  • Clear comments explaining WHY, not just WHAT
  • Encourage users to modify and break them
Progressive Complexity
  • Pong: ~200 lines, 1 system, basic collision
  • Platformer: ~500 lines, 3-4 systems, intermediate
  • Shooter: ~800 lines, 5-6 systems, entity management
  • Advanced: ~1500+ lines, custom everything
Real Games, Not Toys
  • Actually fun to play (even if simple)
  • Complete game loop (start, play, win/lose, restart)
  • Polished enough to be satisfying
  • Demonstrate best practices

📖 Documentation Strategy

Learning Path Documentation
  • Getting Started: "Run Pong in 5 minutes"
  • Tutorial 1: "Modify Pong to add power-ups" (Layer 2)
  • Tutorial 2: "Build a simple game from scratch" (Layer 2-3)
  • Tutorial 3: "Optimize your game loop" (Layer 3-4)
  • Advanced: "Custom render pipeline" (Layer 4)
API Reference By Layer
  • Clearly mark which API layer each function belongs to
  • Show simpler alternatives for advanced functions
  • Link to demo code that uses each API

🚧 Implementation Notes

Current Status: VISION PHASE - NOT IMPLEMENTED YET

This is future work. Right now we're focused on getting the logger working properly with the existing engine structure.

Why Document Now? We know where the tech debt is. We know how we want to evolve the API. Writing this down now means we won't forget the vision when we're ready to implement it.

Technical Challenges To Solve Later

  • Wrapper Implementation: How do high-level APIs wrap low-level without performance cost?
  • Memory Management: Who owns what at each layer? Clear ownership rules needed.
  • Error Handling: Layer 1 might hide errors; Layer 4 exposes everything. How to balance?
  • Build System: How to package demos efficiently? Asset management?
  • Documentation Generation: Auto-generate API docs that clearly show layers?

Open Questions

  • Should Layer 1 have a config-free version? (engine.justRun()?)
  • How to handle save/load at different layers?
  • Should there be a Layer 0 for absolute beginners? (GUI builder?)
  • How do we version the API as layers evolve?
  • Asset pipeline - same progressive disclosure model?

✅ Success Criteria

We'll Know This Works When:
  • A complete beginner can run zig build run-pong and see Pong in 30 seconds
  • That same beginner can read pong/main.zig and understand 80% of it
  • An intermediate developer can customize Pong in an hour
  • A senior developer can build a custom game with Layer 4 and not feel constrained
  • Each demo game is actually fun to play for at least 2-3 minutes
  • Users can transition smoothly between layers as they learn

Remember

Shitty versions right now at all levels, but legit overtime.

Every great engine started somewhere. Unity's early APIs were rough. Unreal evolved through multiple major rewrites. Our job isn't to build the perfect engine on day one - it's to build something that works, teaches well, and improves iteratively.

The demos will be simple at first. The APIs will have rough edges. The documentation will grow over time. And that's perfectly fine. Ship it, learn from it, make it better.
