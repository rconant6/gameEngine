🛠️ Ecosystem Architecture — The Full Picture

North Star: Click an app. Build your art, design your game, click Publish. Walk away with an executable that runs on your target platform. Easy enough for a motivated teenager to make their first game. Powerful enough for the person who built it to keep going deeper.

How We Got Here

Started with Asteroids — felt the raw pain of doing everything by hand. Built ZixelArt to solve the pixel art problem. ZixelArt needed a color system, so that got built. The color system needed a UI, and now the UI needs a real framework. Each layer was justified by the pain of the layer below it. No speculative abstraction — earned every piece.

The destination has finally become clear: a Unity-inspired dev suite where the editor and the runtime are separate things that communicate through a defined contract. You build in the DevApp. You ship the GameRuntime.

🏗️ The Full Stack

Two distinct products share a common foundation:

🖥️ DevApp

The authoring environment. What you use to build games.

  • Hosts all tools (ZixelArt, GameEditor, Exporter…)
  • Owns ProjectState — the shared asset registry
  • Runs on: Platform + Renderer + UI
  • Uses engine only for preview/simulation

🎮 GameRuntime

The shipped player. What the game maker's players run.

  • Standalone executable, no dev tools
  • Consumes what DevApp produced (assets, scenes)
  • Runs on: Platform + Renderer + Engine
  • Knows nothing about DevApp or editors
DevApp authoring only Tool host, tool switching, ProjectState, shared services (asset loader, file watcher). The "Unity Editor" equivalent. Owns the window lifecycle and routes input to the active tool.
↕
Tools authoring only ZixelArt (pixel editor), GameEditor (scene/entity editor), Exporter (build pipeline). Each tool owns its own state and UITree. Tools communicate only through ProjectState — never directly with each other.
↕
src/ui authoring only Node tree, layout engine, input dispatch, command queue. Comptime-generic over app state — zero cost, full type safety. Immediate mode traversal with tree structure for routing.
↕
src/engine runtime primary Game loop, entity system, physics, audio. Ships inside the exported game executable. DevApp uses it only for in-editor preview/simulation. The two products share code but diverge at the boundary.
↕
src/renderer shared Draw calls, GPU state, shader pipeline, RenderContext (including default font). The single convergence point — everything above resolves to draw calls here. Knows nothing about UI, tools, or game state.
↕
src/platform shared Window, input (raw + semantic transitions), file system, timing. Provides InputFrame with just_pressed/just_released/held per button. Both DevApp and GameRuntime start here.
The contract between DevApp and GameRuntime is the asset/data format — what ZixelArt writes out, what the scene editor serializes, what the exporter packages. Both sides agree on the format. Neither leaks into the other.

📐 The UI System — src/ui

The UI system is the piece that makes building tools not painful. Standalone — zero engine deps. Three layers built on top of each other. Full design doc →

Three Layers

Layer 1 — Layout (current work)
Box / Sizing / Resolver. Pure math, zero deps. Describe a tree, get back a flat list of named rects. src/ui/main.zig is the API spec — if it compiles with no renderer import, the layer is correct.
Layer 2 — Widgets
ui.button(), ui.panel(), ui.label(). Takes a ResolvedTree + input state, returns bool/value. Knows about renderer (to draw states). Does not know about tool state.
Layer 3 — Behavior
Per-tool. SaveButton knows what save means in zixelart. ColorSwatch knows it mutates active_color. Built on Layer 2. This is where tool logic lives.
Platform as peer, not layer
Platform supplies raw input (mouse x/y, buttons) and window size. Flows DOWN through all layers. Never owns them. Renderer receives draw calls from Layer 2 — flows UP. Both are infrastructure, not layers.

The data flow

describe (Box tree) → resolve (Resolver: measure + place passes) → []ResolvedNode { name, rect } → Layer 2 widgets consume rects + input → draw + return bool → Layer 3 behavior consumes bool → does the thing Platform feeds window size into resolve(). Platform feeds mouse/keys into Layer 2/3. Renderer receives draw calls from Layer 2.

File structure

src/ui/ ui.zig ← public surface, re-exports everything main.zig ← API design surface / living spec (imports ui only) Sizing.zig ← fixed(n) / grow / fit Box.zig ← leaf / vstack / hstack + Config Resolver.zig ← measure pass + place pass → ResolvedTree Rect.zig ← 2D rect math (exists, stable) Node.zig ← vtable widget node (exists, used by zixelart today) UITree.zig ← flat node tree driver (exists, used by zixelart today) Event.zig ← input event types (exists) Layout.zig ← anchor/split helpers (exists)

🗂️ State Hierarchy

One source of truth at each level. Tools communicate through ProjectState, not directly with each other.

ProjectState ← DevApp owns. All tools can read. The asset registry. name: []const u8 // "MyAsteroidGame" sprites: []SpriteAsset // what ZixelArt produces scenes: []Scene // what GameEditor produces config: GameConfig // resolution, title, target platform, controls └── ZixelState ← ZixelArt owns. Private working state. └── EditorState ← GameEditor owns. Private working state. └── ... // Tools write to ProjectState on save. // Tools read from ProjectState on open. // Tools never call into each other directly.

🚦 Known Rub Points

Platform input gap — currently exposes isPressed (level-triggered). UI system needs just_pressed / just_released transitions and eventually double_click. Platform layer needs to compute and expose these. Not hard, but required before the UI event system can be built properly.
Font in RenderContext — currently threaded as a separate parameter through every render call. Should move into RenderContext as default_font. Eliminates the separate parameter from every on_render signature.
Re-entrant tree mutation — callbacks can't modify the tree while it's being traversed. Deferred command queue solves this, but needs to be designed before the first dynamic node (modal, tooltip, context menu) is needed.
App is currently ambiguous — src/app/App is currently "a thin game loop wrapper." It needs to split into: DevApp shell (tool host) and a minimal runtime bootstrap that GameRuntime uses. These are different things wearing the same name right now.

🗺️ Build Order

Each step is justified by concrete pain from the step before it. See UI Framework doc for Layer 1 detail.

  1. UI Layer 1 — Layout engine — Sizing, Box, Resolver. Pure math, zero deps. src/ui/main.zig compiles with no renderer import. uitest visualizes resolved rects. In Progress
  2. UI Layer 2 — Widgets — button(), panel(), label(). Built on Layer 1 + renderer. Returns bool/value, no tool knowledge. Next
  3. ZixelArt migration — retrofit zixelart panels onto the real UI framework. Validate against real pain. Layer 3 behavior wired per-widget. Soon
  4. RenderContext font — move default_font into RenderContext. Clean up all call sites. Soon
  5. src/app DevApp shell — tool switching, ProjectState, shared services. ZixelArt becomes the first registered tool. Later
  6. ZEditor tool — scene/entity editor, uses engine for preview. Second tool in the DevApp. Later
  7. Exporter — build pipeline, platform targets, produces the standalone GameRuntime executable. The "Publish" button. Later

🎯 The End-to-End Flow

Developer opens DevApp │ ├── opens ZixelArt │ ├── draws sprites using palette + harmony system │ ├── saves → writes SpriteAsset to ProjectState │ ├── opens GameEditor │ ├── places sprites in scenes │ ├── configures entities, physics, controls │ ├── previews using GameRuntime (sandboxed, in-editor) │ ├── saves → writes Scene to ProjectState │ ├── clicks Publish │ ├── Exporter reads ProjectState │ ├── packages: assets + scenes + config + GameRuntime binary │ ├── compiles for target platform (mac, windows, wasm) │ └── outputs: MyGame.app / MyGame.exe / MyGame.html │ └── hands it to a 16-year-old └── it runs
The philosophy that got us here: Feel the pain first, then abstract. Every layer in this stack was earned by the friction of the layer below it. No speculative architecture — just the minimum needed to solve the real problem in front of you, designed to extend cleanly when the next real problem shows up.
