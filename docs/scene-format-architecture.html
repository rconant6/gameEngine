Overview

The scene format is a custom declarative DSL for defining game scenes, entity templates, and asset declarations. It uses indent-based nesting (2 spaces, no tabs) and bracket-based declaration headers. The system has two halves:

Language & Parsing (scene-format module) src/scene-format/ // zero external dependencies
  token.zig, lexer.zig, parser.zig, ast.zig
Runtime (scene module) src/scene/ // depends on ECS, renderer, assets, registry, action
  manager.zig, templates.zig, instantiator.zig, loader.zig

Language Reference

Quick syntax summary for the .scene and .template format.

Indentation 2 spaces per level. Tabs are forbidden. Indent/dedent generates tokens.
Comments // line comments (anywhere on a line)
Declarations [Name:type] where type = scene | entity | template | asset
Components [ComponentName] or [ComponentName:shape_type]
Properties name:type value
Types f32, i32, u32, bool, string, color, vec2, vec3, asset_ref
Arrays type[] for array annotation, {val1, val2, ...} for array values
Vectors {x, y} for vec2, {x, y, z} for vec3
Colors #RRGGBB or #RRGGBBAA (hex)
Strings Double-quoted: "text here"
Asset References asset_ref "AssetName" (refers to a declared asset by name)

Minimal Example

[MyScene:scene]
  [Ball:entity]
    [Transform]
      position:vec2 {0.0, 0.0}
    [Sprite:circle]
      radius:f32 5.0
      fill_color:color #ff0000

Declaration Types

Top-level declarations use the [Name:type] syntax. Four types are supported.

[Name:scene]
Container for entities and nested scenes. Can nest arbitrarily.
Children: entity, scene, asset declarations
[GameWorld:scene]
  [Player:entity]
    // components...
  [UI:scene] // nested scene
    [HUD:entity]
      // components...
[Name:entity]
Defines an entity with component blocks as children.
Creates an entity in the World with each component added via addComponent().
[Name:template]
Reusable entity blueprint. Same structure as entity but loaded into TemplateManager.
Typically in separate .template files. Spawned at runtime via createEntityFromTemplate().
Template names are case-insensitive for lookup.
[Name:asset type]
Declares an asset to load. Currently only font type is supported.
The Name becomes the asset reference identifier used by components.
[GameFont:asset font]
  filename:string "Orbitron.ttf"

[TitleFont:asset font]
  abs_path:string "assets/fonts/title.ttf"

[SystemFont:asset font]
  path:string "assets/fonts/"
  filename:string "arial.ttf"

Component Blocks

Inside entity and template declarations. Three variants based on the component type.

Generic Components

Most components use the plain [ComponentName] syntax with typed properties.

[Transform]
  position:vec2 {5.0, -3.0}
  rotation:f32 0.0
  scale:f32 1.0

[Velocity]
  linear:vec2 {3.0, 0.0}
  angular:f32 0.0

[Tag]
  tags:string "player,ship"

[Lifetime]
  remaining:f32 2.0

[Text]
  text:string "Hello World"
  font_asset:asset_ref "GameFont"
  text_color:color #FF8800
  size:f32 0.8

Sprite Components (shape-typed)

Sprites use [Sprite:shape_type] syntax. The shape type determines which geometry fields are expected.

[Sprite:circle]
  origin:vec2 {0.0, 0.0}
  radius:f32 2.0
  fill_color:color #00FF00
  stroke_color:color #FFFFFF
  stroke_width:f32 1.0
  visible:bool true
[Sprite:rectangle]
  center:vec2 {0.0, 0.0}
  half_width:f32 1.5
  half_height:f32 1.0
  fill_color:color #AA00FF
[Sprite:triangle]
  v0:vec2 {0.0, 1.0}
  v1:vec2 {-1.0, -1.0}
  v2:vec2 {1.0, -1.0}
  fill_color:color #0000FF
[Sprite:polygon]
  points:vec2[] {
    {0.0, 1.0},
    {0.951, 0.309},
    {0.588, -0.809},
    {-0.588, -0.809},
    {-0.951, 0.309}
  }
  fill_color:color #00FFAA

Collider Components (shape-typed)

Same [Collider:shape_type] syntax. Shape properties match the collider shape.

[Collider:circle]
  radius:f32 1.0

[Collider:rectangle]
  center:vec2 {0.0, 0.0}
  half_width:f32 1.5
  half_height:f32 1.8

Properties & Values

Properties follow the pattern: name:type value. The type annotation tells the parser how to interpret the value.

Type Annotations

Numeric f32 // floating point: 1.0, -3.5, 0.0
i32 // signed integer: 0, -1, 42
u32 // unsigned integer: 0, 1, 255
Vectors vec2 // 2D vector: {x, y}
vec3 // 3D vector: {x, y, z}
Other bool // boolean: true, false
string // quoted string: "hello"
color // hex color: #RRGGBB or #RRGGBBAA
asset_ref // asset name reference: "FontName"
Arrays Append [] to any type: vec2[], f32[], string[]
Array values: {val1, val2, val3} with optional multiline

Value Examples

// Scalars
speed:f32 3.5
health:i32 100
damage:f32 -1.5 // negative numbers supported
visible:bool true

// Strings
name:string "Player One"

// Colors
fill_color:color #FF0000 // red, opaque
bg_color:color #FF000080 // red, 50% alpha

// Vectors
position:vec2 {5.0, -3.0}
position3d:vec3 {0.0, 0.0, 1.0}

// Asset references
font_asset:asset_ref "OrbitronFont"

// Arrays (inline)
points:vec2[] {{0.0, 1.0}, {1.0, 0.0}, {-1.0, 0.0}}

// Arrays (multiline)
points:vec2[] {
  {0.0, 0.5},
  {0.15, 0.2},
  {-0.15, 0.2}
}

Triggers & Actions

[OnInput] and [OnCollision] components use nested [trigger] and [action] blocks.

OnInput (keyboard/mouse triggers)

[OnInput]
  [trigger]
    key:string "Space"
    [action]
      type:string "spawn_entity"
      template_name:string "Projectile"
      offset:vec2 {0.0, 2.0}
      priority:i32 0
  [trigger]
    key:string "W"
    [action]
      type:string "set_velocity"
      target:string "self"
      velocity:vec2 {0.0, 5.0}
      priority:i32 0

OnCollision (tag-based triggers)

[OnCollision]
  [trigger]
    other_tag_pattern:string "enemy"
    [action]
      type:string "spawn_entity"
      template_name:string "Explosion"
      offset:vec2 {0.0, 0.0}
      priority:i32 0
    [action]
      type:string "destroy_self"
      priority:i32 1
    [action]
      type:string "destroy_other"
      priority:i32 2

Action Types

destroy_self // mark self for destruction
  priority:i32

destroy_other // mark colliding entity for destruction
  priority:i32

spawn_entity // create entity from template at collision point
  template_name:string   offset:vec2   priority:i32

set_velocity // change velocity of self or other
  target:string // "self" or "other"   velocity:vec2   priority:i32

debug_print // log a message
  message:string   priority:i32

play_sound // play audio (stub)
  sound:string   priority:i32

Priority: Lower numbers execute first. Multiple actions in a trigger execute in priority order.

Lexer

Tokenizes source text. Indent-aware: tracks 2-space indentation and emits INDENT/DEDENT tokens. src/scene-format/lexer.zig

Lexer
Fields src: [:0]const u8 // null-terminated source
index: u32 // current position
token_start: u32
src_loc: SourceLocation // line/col tracking
indent_level: u32 // current indent (in 2-space units)
dedents_left: u32 // pending dedent tokens
space_count: u32 // spaces at line start
hex_color_count: u32
API init(src) → Lexer
next() → !?Token // null at EOF

Indentation Rules

Exactly 2 spaces per indentation level
Tabs are forbidden (LexerError.TabIndentationFound)
At each new line: compare new indent level vs current
  Increased by 1 level → emit INDENT token
  Decreased by N levels → emit N DEDENT tokens (one per call to next())
  Same level → no indent/dedent tokens
Odd space counts cause InvalidIndentation error

Lexer Errors

InvalidCharacter // unrecognized character
InvalidColor // # not followed by 6 or 8 hex digits
InvalidIndentation // odd number of spaces
InvalidNumber // malformed number literal
TabIndentationFound // tab character at line start
UnclosedString // missing closing quote

Parser

Recursive descent parser. Consumes tokens from the Lexer and builds the AST. src/scene-format/parser.zig

Parser
Fields allocator: Allocator
lexer: Lexer
current_tok: Token
previous_tok: ?Token
file_name: []const u8
API init(alloc, src, file_name) → Parser
parse() → SceneFile

Grammar

SceneFile      := Declaration*
Declaration    := "[" identifier ":" DeclType "]" DeclBody

DeclType       := "scene" | "entity" | "template" | "asset" AssetType
AssetType      := "font"

DeclBody(scene)    := INDENT Declaration* DEDENT?
DeclBody(entity)   := INDENT ComponentBlock* DEDENT?
DeclBody(template) := INDENT ComponentBlock* DEDENT?
DeclBody(asset)    := INDENT Property* DEDENT?

ComponentBlock     := "[" identifier (":" shape_type)? "]" CompBody
CompBody           := INDENT (Property | NestedBlock)* DEDENT?

NestedBlock        := "[" identifier "]" INDENT (Property | NestedBlock)* DEDENT

Property           := identifier ":" TypeAnnotation Value
TypeAnnotation     := BaseType ("[" "]")?

Value              := ScalarValue | ArrayValue
ScalarValue        := VectorValue | number | "-" number | boolean | string_lit | color_lit
VectorValue        := "{" number ("," number)+ "}"
ArrayValue         := "{" INDENT? (ScalarValue ("," ScalarValue)*)? DEDENT? "}"

Parser Errors

UnexpectedToken // wrong token at position
UnknownDeclarationType // not scene/entity/template/asset
UnknownType // unrecognized type annotation
UnknownAssetType // not "font"
UnknownComponentType // Sprite/Collider with unknown shape
UnsupportedVectorLength // vec with wrong number of elements
UnclosedArrayBrackets // missing } for array
NoPreviousTokens / NoTokenReturned
Unimplemented
+ all LexerError types propagate

Value Extraction (type conversion)

The extractValueForType function maps AST Values to Zig types:

f32 ← Value.number (cast from f64)
i32 / u32 ← Value.number (integer cast)
bool ← Value.boolean
[]const u8 ← Value.string

End-to-End Pipeline

How a .scene file goes from text on disk to running entities in the game.

Loading & Parsing

  1. engine.loadScene("game", "game") → SceneManager calls Loader
  2. Loader reads assets/scenes/game.scene from disk (max 1MB)
  3. Lexer tokenizes source: handles indentation, keywords, literals, comments
  4. Parser builds AST: SceneFile with Declaration tree
  5. SceneManager stores the SceneFile by name

Template Loading

  1. engine.loadTemplates("assets/templates") → TemplateManager
  2. Scans directory for all .template files
  3. Each file is loaded and parsed identically to scenes
  4. Each [Name:template] block registered in template map (lowercase key)

Activation & Instantiation

  1. engine.setActiveScene("game") → marks scene as active
  2. engine.instantiateActiveScene() → Instantiator.instantiate(scene_file)
  3. Walk AST declarations top-to-bottom:
  4.   Asset declarations → load fonts, register in AssetManager
  5.   Scene declarations → recurse into child declarations
  6.   Entity declarations → world.createEntity(), then for each component:
  7.     Generic: reflection-match properties to struct fields → world.addComponent()
  8.     Sprite: build shape geometry + colors → world.addComponent(entity, Sprite, ...)
  9.     Collider: build collider shape → world.addComponent(entity, Collider, ...)
  10.     OnInput/OnCollision: build nested triggers + actions → world.addComponent()
  11. All created entities tracked in last_instantiated_entities for cleanup on reload

Runtime Spawning

  1. ActionExecutor encounters spawn_entity action with template name
  2. world.createEntityFromTemplate(name, position)
  3. TemplateManager looks up template (case-insensitive)
  4. Instantiator creates entity, adds components from template declaration
  5. Transform.position receives the spawn position offset

Scene Reload

  1. engine.reloadActiveScene()
  2. Destroy all entities from last_instantiated_entities
  3. Re-read .scene file from disk (picks up edits)
  4. Re-parse and re-instantiate
