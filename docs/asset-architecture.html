Overview

The asset system currently manages fonts (TrueType parsing + vector glyph rendering) and defines the ZXL pixel art format (for ZixelArt). Assets are loaded lazily, deduplicated by path, and referenced via opaque handles.

Deduplication

Before loading a font, the manager normalizes the path and checks path_to_handle. If the font is already loaded, the existing handle is returned. Embedded fonts use synthetic paths like <embedded:orbitron>.

TrueType Parsing

The Font constructor parses a TrueType (.ttf) binary file. All data is big-endian. src/assets/font/font.zig + src/assets/font/font_data.zig

Tables Read

  1. Font Directory — locate all tables by tag (head, hhea, hmtx, maxp, cmap, loca, glyf)
  2. head — units_per_em, index_to_loc_format (short/long loca offsets), checksum validation
  3. maxp — num_glyphs (total glyph count)
  4. hhea — ascender, descender, line_gap, number of horizontal metrics
  5. hmtx — per-glyph advance_width and left side bearing
  6. cmap — character-to-glyph mapping (Format 4 subtable, Unicode platform)
  7. loca — glyph offsets into glyf table (short or long format)
  8. glyf — glyph outlines (contours, points, flags)

Key Binary Structures

Font Directory
sfnt_version: u32 // 0x00010000 for TrueType
num_tables: u16
Followed by TableEntry[num_tables]: { tag: u32, checksum: u32, offset: u32, length: u32 }
HeadTable
units_per_em: u16 // design units (typically 1000 or 2048)
index_to_loc_format: i16 // 0 = short (u16), 1 = long (u32)
magic_number: u32 // must be 0x5f0f3cf5
CmapFormat4
Segment-based mapping for BMP characters (U+0000 to U+FFFF)
seg_countx2: u16 // 2 * number of segments
Arrays: endCode[], startCode[], idDelta[], idRangeOffset[]
For char in [startCode..endCode]: glyph = char + idDelta (or via rangeOffset)
GlyphFlag (packed bitfield)
on_curve: u1 // 1 = on outline, 0 = control point (quadratic bezier)
xShort: u1 // X delta is 1 byte   yShort: u1 // Y delta is 1 byte
repeat: u1 // next byte is repeat count
x_same_or_pos: u1   y_same_or_pos: u1

CmapTableNotFound, GlyfTableNotFound, LocaTableNotFound

+ corrupted checksums for each table

Glyph Pipeline

How a character goes from TrueType binary to rendered triangles on screen.

  1. Parse glyf table — extract contour endpoints, point coordinates (delta-encoded), and flags from binary
  2. Filter to on-curve points — discard quadratic bezier control points, keep only on-curve vertices. Normalize coordinates to 0.0–1.0 range. Store as FilteredGlyph
  3. Build triangles (lazy, on first render) — glyph_builder.buildTriangles():
      a) Remove duplicate consecutive vertices (epsilon < 0.00001)
      b) Remove collinear points (sin(angle) < 0.01)
      c) Ear-clip polygon to triangles
  4. Cache triangles — store in Font.glyph_triangles map for reuse
  5. Render — text.zig reads triangles, applies position/scale, submits as geometry draw calls

EarClipper

EarClipper
vertices: []const Point // all glyph points
polygon: ArrayList(usize) // active vertex indices
reflex: ArrayList(VertexClass) // convex/reflex classification
triangles: ArrayList(Triangle) // output
winding_sign: f32 = -1.0 // -1=CW, +1=CCW
Algorithm triangulate(contours) → []Triangle
1) Classify contours by winding (CW = outer, CCW = holes)
2) Bridge holes into outer contour via rightmost vertex
3) Iteratively clip ear tips (convex verts with no interior reflex verts)
4) Handle concave shapes by adapting winding when remainder flips

ZXL Format

The engine's custom pixel art format for ZixelArt. Palette-indexed, multi-frame, compact binary. src/zxl/

Binary Layout

OffsetSizeTypeDescription
04u8[4]Magic: "ZXL\x00"
41u8Version (currently 1)
51u8Flags (reserved)
62u16 LEFrame count
82u16 LECanvas width
102u16 LECanvas height
121u8Palette size (0 = 256)
133u8[3]Reserved (zeroed)
16N×4Rgba[N]Palette colors (R,G,B,A per entry)
16+N×4varFrame[]Frame data (see below)

Frame Layout

FieldSizeTypeDescription
name_len1u8Length of frame name
namename_lenu8[]Frame name (UTF-8)
duration_ms2u16 LEAnimation frame duration
origin_x2i16 LESprite pivot X
origin_y2i16 LESprite pivot Y
reserved1u8Reserved
pixelsW×Hu8[]Palette indices (row-major)

Design notes: All multi-byte values are little-endian. Palette index 0 is always transparent by convention. Max 256 colors per image. Frame pixels are palette indices, not raw RGBA — very compact for pixel art.

ZXL Format Reader / Writer

ZXL Reader / Writer

Binary serialization and deserialization for the ZXL format. src/zxl/ZxlReader.zig + src/zxl/ZxlWriter.zig

ZxlReader
fromBytes(alloc, data) → ZxlImage // deserialize from byte slice
fromFile(alloc, path) → ZxlImage // read file + deserialize
ZxlWriter
toBytes(alloc, image) → []u8 // serialize to byte slice
toFile(alloc, image, path) // serialize + write to disk

Reader Flow

  1. Validate magic bytes ("ZXL\x00") and version (must be 1)
  2. Read header: frame count, canvas dimensions, palette size
  3. Read palette: N × 4 bytes (RGBA per color)
  4. For each frame: read name (length-prefixed), duration, origin, then pixel data (W×H palette indices)
  5. Return populated ZxlImage

Writer Flow

  1. Write magic, version, flags
  2. Write frame count, canvas dimensions, palette size
  3. Write palette colors
  4. For each frame: write name (length-prefixed), duration, origin, pixel data

Errors

InvalidMagic // wrong file signature
UnsupportedVersion // version != 1
UnexpectedEof // file truncated
InvalidPixelData // frame pixel count != width * height

Scene Asset Loading

How assets declared in .scene files get loaded into the AssetManager. src/scene/instantiator.zig

Scene File Declaration

[GameFont:asset font]
  filename:string "orbitron.ttf"

[TitleFont:asset font]
  abs_path:string "/usr/share/fonts/title.ttf"

Loading Flow

  1. Instantiator encounters AssetDeclaration with type .font
  2. Extract properties: filename, path, or abs_path
  3. Load via one of three methods:
      a) abs_path → AssetManager.loadFontFromPath(abs_path)
      b) filename + path → loadFontFromPath(path/filename)
      c) filename only → loadFont(filename) (uses base font_path)
  4. AssetManager.registerFontAsset(asset_name, handle) — bind scene name to handle
  5. Later, Text components reference fonts by handle: font_asset: FontHandle

AST Types

AssetDeclaration { name, asset_type: AssetType, properties: ?[]Property }
AssetType = enum { font } // + texture, audio, shader, sprite, material, mesh planned
Property { name, type_annotation, value }
Value = number | string | color | bool | vector | assetRef | array

Asset Lifecycle

Complete flow from build time to shutdown.

1. Build Time

@embedFile("default_orbitron.ttf") compiled into executable
Asset module linked with math + renderer dependencies

2. Engine Initialization

  1. AssetManager.init(allocator)
  2. FontManager.init() with base path "assets/fonts/"
  3. Load embedded Orbitron from memory → registered as "__default__"
  4. Default FontHandle always available

3. Scene Load Time

  1. SceneManager.loadScene(name, path) → parse .scene file to AST
  2. Instantiator.instantiate(scene_file) walks declarations
  3. For each AssetDeclaration: load TTF, parse tables, build glyph shapes
  4. Register in name_to_font map by scene asset name
  5. For each EntityDeclaration with Text component: store FontHandle reference

4. Runtime Rendering

  1. RenderSys queries Transform + Text
  2. Retrieve Font via AssetManager.getFont(text.font_asset)
  3. For each character: map to glyph → lazy-triangulate → cache → draw
  4. Advance cursor by advance_width * scale

5. Shutdown

  1. AssetManager.deinit()
  2. FontManager.deinit() — for each Font: free glyph_shapes, glyph_triangles, char_to_glyph, advance_widths
  3. Free path_to_handle and name_to_font hashmaps

Current Limitations

Fonts only — no sprite/texture/audio assets yet
No async loading — all synchronous on main thread
No hot-reload — assets loaded once, immutable
No resource pooling — each load is independent
Single-threaded — not thread-safe for concurrent loads
Glyph triangles computed per-Font instance (not shared across same font at different sizes)
