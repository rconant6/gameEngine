Overview

A custom ECS built on sparse-dense array storage for O(1) add/remove/lookup and cache-friendly iteration. Type-erased storages allow the World to hold any component type. Queries are built at comptime with generic tuples.

Type Erasure

The World stores all component storages in a single StringHashMap keyed by type name. Each storage is wrapped in a StorageInterface with a VTable for type-erased operations (remove, has, destroy). This allows the World to iterate all storages when destroying an entity without knowing concrete types.

Entity

A simple value type — just an ID. No methods, no inheritance. src/ecs/Entity.zig

World Query Components

Query

Comptime-generated iterator that efficiently finds entities matching a set of component types. src/ecs/Query.zig

Query(ComponentTypes...)
storages: ComponentTypes // tuple of storage pointers
primary_index: usize // index of primary storage (index 0)
current_index: usize // iteration position
Iteration next() → ?Entry
Query Entry
entity: Entity
components: ComponentPointers // tuple of *T pointers
get(comptime index) → *T // get component by tuple index

Query Algorithm

  1. Iterate the primary storage's dense array (first component type)
  2. For each entity_id in primary storage, check that all other storages also have that entity
  3. If all match, return an Entry with mutable component pointers
  4. If any storage doesn't have the entity, skip and continue

Usage Pattern

var query = world.query(.{ *Transform, *Velocity });
while (query.next()) |entry| {
  const transform = entry.get(0); // *Transform
  const velocity = entry.get(1); // *Velocity
  transform.position = transform.position.add(velocity.linear.mul(dt));
}

CameraTrackingSys (spring-damper)

Queries CameraTracking + ActiveCamera
Immediate mode: snap camera to target + offset
Smooth mode: spring_force = displacement * stiffness

Systems

Pure functions called by the engine each frame. Each system queries specific component combinations and operates on them. src/systems/

Execution order (per frame):

  1. ActionSys — process input/collision triggers, execute queued actions
  2. PhysicsSys — apply forces, update velocities (stub)
  3. MovementSys — position += velocity * dt
  4. CollisionDetectionSys — detect overlaps, populate collision events
  5. CameraTrackingSys — spring-damper follow target entity
  6. LifetimeSys — decrement timers, mark expired entities with Destroy
  7. RenderSys — draw world-space sprites/text, then screen-space UI
ActionSys
run(world, action_system, trigger_ctx)
1) Process all registered trigger systems (input, collision)
2) Execute queued actions via ActionExecutor
3) Clear queue for next frame
MovementSys

PhysicsSys

CollisionDetectionSys

Populates Collision events with point, normal, penetration
CameraTrackingSys

Entity Lifecycle

How entities are born, live, and die in the ECS.

Creation

  1. world.createEntity() → increment next_entity_id, return Entity{ .id = id }
  2. world.addComponent(entity, T, value) → auto-register type if first use, add to ComponentStorage(T)
  3. Repeat for each component the entity needs

From Template

  1. world.createEntityFromTemplate(name, offset)
  2. TemplateManager looks up template by lowercase name
  3. Instantiator creates entity and adds all components from template definition
  4. Transform.position gets the offset added

From Scene File

  1. instantiator.instantiate(scene_file) walks the AST
  2. For each entity declaration: create entity + build components from properties
  3. For each asset declaration: load fonts (and eventually textures, audio)
  4. For each scene declaration: recursively instantiate nested scenes

Destruction

  1. Entity gets Destroy marker component (via LifetimeSys, ActionExecutor, or manual)
  2. LifetimeSys cleanup pass: query Destroy → call world.destroyEntity(entity)
  3. destroyEntity iterates all storages, removes entity from each
  4. Each ComponentStorage.remove() calls deinit() if the component type has one (with or without allocator param)
  5. Swap-with-last in dense array to maintain packing

Data Flow

Key data paths through the ECS during a frame.

Query Flow

world.query(.{Transform, Velocity})
→ Register component types if needed
→ Get storages for each type from component_storages map
→ Create Query with storage tuple
→ Iterate: primary_storage.dense → check other storages → yield mutable pointers

Collision → Action Flow

CollisionDetectionSys: query(Transform, Collider) → test pairs → Collision events
↓
CollisionTrigger.process(): for each Collision → match entity tags → enqueue Actions
↓
ActionExecutor: sort by priority → execute (destroy, spawn, set_velocity...)

Input → Action Flow

Platform events → Input state (isPressed/isDown/isReleased)
↓
InputTrigger.process(): query(OnInput) → check input state → enqueue Actions
↓
ActionExecutor: execute queued actions on World

Scene Loading Flow

                  assets → loadFont, register handle
                  scenes → recurse
↓
Entities live in World, ready for systems to process
