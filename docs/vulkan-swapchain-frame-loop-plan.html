Vulkan Renderer — Swapchain & Frame Loop

Phase 6 of the rendering stack. Instance → Surface → PhysicalDevice → Device are done. This plan adds everything needed to get pixels on screen: a swapchain, render pass, framebuffers, command recording infrastructure, and a working beginFrame/endFrame loop that clears to a color and presents.

Goal: A window that clears to a solid color each frame and presents it via the swapchain. No draw calls, no pipeline, no shaders — just the frame machinery. Once this works, all future rendering (geometry, textures, UI) slots into beginFrame/endFrame without touching the infrastructure again.

Phase 1

Device Extension modify

Device.zig currently sets enabledExtensionCount = 0. Swapchain creation will fail at runtime without VK_KHR_swapchain as a device extension. This is the only change to existing files before the new modules are built.

Device.zig changes

  • Add extension list: const device_extensions = [_][*c]const u8{ "VK_KHR_swapchain" };
  • Set dev_create_info.enabledExtensionCount = device_extensions.len;
  • Set dev_create_info.ppEnabledExtensionNames = &device_extensions;
  • Confirm build still succeeds and engine still launches
Why device extension, not instance? VK_KHR_surface and VK_KHR_wayland_surface are instance extensions — they teach the instance how to create a surface object. VK_KHR_swapchain is a device extension — it teaches the device how to present images to that surface. Different layers of the Vulkan object model.
Phase 2

Swapchain new

The swapchain is a ring of GPU images the compositor cycles through. You render into one while the compositor is displaying another. This module queries what the surface supports, picks sensible defaults, and creates the swapchain + per-image views.

Swapchain.zig — struct fields

  • handle: *vk.struct_VkSwapchainKHR_T
  • images: []vk.VkImage — allocated with gpa, freed in deinit
  • views: []vk.VkImageView — one per image, destroyed in deinit
  • format: vk.VkFormat — chosen surface format
  • extent: vk.VkExtent2D — chosen image dimensions

init() — query surface capabilities

  • vkGetPhysicalDeviceSurfaceCapabilitiesKHR → caps
  • Image count: caps.minImageCount + 1, clamped to caps.maxImageCount if non-zero
  • Extent: use caps.currentExtent if not 0xFFFFFFFF; otherwise clamp requested width/height to min/maxImageExtent
  • Pre-transform: caps.currentTransform

init() — pick surface format

  • vkGetPhysicalDeviceSurfaceFormatsKHR (two-call pattern: get count, then fill slice)
  • Prefer VK_FORMAT_B8G8R8A8_SRGB + VK_COLOR_SPACE_SRGB_NONLINEAR_KHR
  • Fall back to formats[0] if preferred not found
  • Store chosen format.format in self.format

init() — pick present mode

  • vkGetPhysicalDeviceSurfacePresentModesKHR (two-call pattern)
  • Prefer VK_PRESENT_MODE_MAILBOX_KHR — lowest latency without tearing
  • Fall back to VK_PRESENT_MODE_FIFO_KHR — always available, vsync

init() — create VkSwapchainKHR

  • VkSwapchainCreateInfoKHR: surface, minImageCount, imageFormat, imageColorSpace, imageExtent, imageArrayLayers=1, imageUsage=COLOR_ATTACHMENT
  • imageSharingMode = VK_SHARING_MODE_EXCLUSIVE (graphics == present queue, which is already verified in Device.zig)
  • preTransform from caps, compositeAlpha = OPAQUE, presentMode, clipped = VK_TRUE
  • oldSwapchain = null on first create (pass old handle when recreating on resize)
  • vkCreateSwapchainKHR → self.handle

init() — retrieve images + create image views

  • vkGetSwapchainImagesKHR (two-call pattern) → allocate and fill self.images
  • For each image, create VkImageView:
    • viewType = VK_IMAGE_VIEW_TYPE_2D
    • format = self.format
    • components: all four channels = VK_COMPONENT_SWIZZLE_IDENTITY
    • subresourceRange: aspectMask = COLOR, baseMipLevel=0, levelCount=1, baseArrayLayer=0, layerCount=1
  • Allocate self.views slice and store each image view

deinit()

  • Destroy each image view with vkDestroyImageView
  • Free self.views and self.images slices via gpa
  • vkDestroySwapchainKHR — do NOT destroy the images themselves (swapchain owns them)
Do not destroy VkImage handles. Swapchain images are owned by the swapchain. Calling vkDestroyImage on them is undefined behavior and will crash or corrupt GPU state. Only destroy the VkImageView wrappers you created.
Two-call pattern: Many Vulkan enumerate functions work the same way — call once with a null pointer to get the count, allocate, call again to fill. vkGetSwapchainImagesKHR(dev, swapchain, &count, null) then vkGetSwapchainImagesKHR(dev, swapchain, &count, images.ptr). Same for surface formats and present modes.
Phase 3

Render Pass new

A render pass describes how the GPU transitions image memory during rendering. This one has a single color attachment: clear it at the start, store the result, then transition it to a layout the compositor can display.

RenderPass.zig — struct fields

  • handle: *vk.struct_VkRenderPass_T

init(device, format)

  • VkAttachmentDescription for the color attachment:
    • format = swapchain format (passed in)
    • samples = VK_SAMPLE_COUNT_1_BIT
    • loadOp = VK_ATTACHMENT_LOAD_OP_CLEAR — wipe on entry
    • storeOp = VK_ATTACHMENT_STORE_OP_STORE — keep result
    • stencilLoadOp / stencilStoreOp = DONT_CARE — no depth/stencil
    • initialLayout = VK_IMAGE_LAYOUT_UNDEFINED — don't care about prior contents
    • finalLayout = VK_IMAGE_LAYOUT_PRESENT_SRC_KHR — transition for presentation
  • VkAttachmentReference: attachment = 0, layout = VK_IMAGE_LAYOUT_COLOR_ATTACHMENT_OPTIMAL
  • VkSubpassDescription: pipelineBindPoint = GRAPHICS, colorAttachmentCount = 1, pColorAttachments = &color_ref
  • VkSubpassDependency — synchronize external writes before our color attachment write:
    • srcSubpass = VK_SUBPASS_EXTERNAL, dstSubpass = 0
    • srcStageMask = VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT
    • srcAccessMask = 0
    • dstStageMask = VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT
    • dstAccessMask = VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT
  • vkCreateRenderPass → self.handle

deinit()

  • vkDestroyRenderPass(device, self.handle, null)
Why the subpass dependency? Without it, the GPU may start the subpass before previous frames have finished presenting. The dependency tells the driver: wait for any outstanding COLOR_ATTACHMENT_OUTPUT operations before we write to the color attachment. Without this, you get validation errors and potential visual corruption.
Phase 4

Framebuffers new

A framebuffer binds a specific set of image views to a render pass's attachment slots. One framebuffer per swapchain image — the render pass attachment description stays the same, only the backing image changes per frame.

Framebuffer.zig — functions

  • pub fn init(gpa, device, render_pass, views, extent) ![]vk.VkFramebuffer
  • For each image view, create VkFramebufferCreateInfo:
    • renderPass = render_pass
    • attachmentCount = 1, pAttachments = &views[i]
    • width / height from extent
    • layers = 1
  • Allocate and return a []vk.VkFramebuffer slice (caller owns, gpa allocated)
  • pub fn deinit(gpa, device, framebuffers: []vk.VkFramebuffer) void
  • deinit: call vkDestroyFramebuffer for each entry, then gpa.free(framebuffers)
No struct wrapper needed. A framebuffer slice is simple enough to pass directly. Wrapping it in a struct adds boilerplate for no benefit here — just pass framebuffers[current_image] when beginning the render pass.
Phase 5

Command Pool new

Command buffers are where GPU work is recorded. A pool backs them. One command buffer per frame-in-flight — each frame gets its own buffer so frame N can be recording while frame N-1 is still executing on the GPU.

CommandPool.zig — struct fields

  • pool: *vk.struct_VkCommandPool_T
  • buffers: [frames_in_flight]vk.VkCommandBuffer

init(device, queue_family_index)

  • VkCommandPoolCreateInfo:
    • flags = VK_COMMAND_POOL_CREATE_RESET_COMMAND_BUFFER_BIT — allows individual buffer reset
    • queueFamilyIndex = graphics queue family from PhysicalDevice
  • vkCreateCommandPool → self.pool
  • VkCommandBufferAllocateInfo:
    • commandPool = self.pool
    • level = VK_COMMAND_BUFFER_LEVEL_PRIMARY
    • commandBufferCount = frames_in_flight
  • vkAllocateCommandBuffers → fills self.buffers

deinit(device)

  • vkDestroyCommandPool(device, self.pool, null) — automatically frees all buffers allocated from it
frames_in_flight is a comptime constant. Define it once at the top of CommandPool.zig (or vulkan_renderer.zig) and reference it everywhere. Value: 2. This is the CPU-GPU overlap depth — 2 is the standard sweet spot for most workloads.
Phase 6

Sync Primitives new

Three sync objects per frame-in-flight. Two semaphores for GPU→GPU signaling (acquire → render → present), one fence for CPU to wait on the GPU before reusing this frame's resources.

Sync.zig — FrameSync struct

  • image_available: *vk.struct_VkSemaphore_T — signaled when swapchain image is ready to render into
  • render_finished: *vk.struct_VkSemaphore_T — signaled when command buffer has finished executing
  • in_flight: *vk.struct_VkFence_T — CPU waits on this before reusing this frame's command buffer

init(device) ![frames_in_flight]FrameSync

  • For each frame: create two semaphores with VkSemaphoreCreateInfo{ .sType = ... }
  • Create fence with VkFenceCreateInfo{ .flags = VK_FENCE_CREATE_SIGNALED_BIT }
  • Return array of FrameSync

deinit(syncs: []FrameSync, device)

  • For each frame: vkDestroySemaphore × 2, vkDestroyFence × 1
Fence must start signaled. On frame 0, beginFrame calls vkWaitForFences before any GPU work has been submitted. If the fence starts unsignaled, this blocks forever. VK_FENCE_CREATE_SIGNALED_BIT lets frame 0 pass through immediately.
Why two semaphores per frame? image_available is signaled by vkAcquireNextImageKHR — it tells the GPU "this image is no longer being displayed, you can write to it." It gates the render submission. render_finished is signaled by vkQueueSubmit — it tells the GPU "rendering is done, you can present this image." It gates the present. They cannot be collapsed into one because they signal at different points in the pipeline.
Phase 7

Frame Loop — vulkan_renderer.zig modify

Wire all the new modules into VulkanRenderer and implement the four frame methods. This is the payoff — after this, the engine renders a colored window.

New fields on VulkanRenderer

  • swapchain: Swapchain
  • render_pass: RenderPass
  • framebuffers: []vk.VkFramebuffer
  • cmd: CommandPool
  • sync: [frames_in_flight]Sync.FrameSync
  • clear_color: vk.VkClearColorValue — default to black { .float32 = .{ 0, 0, 0, 1 } }
  • current_frame: u32 = 0
  • current_image: u32 = 0

Extend init()

  • After Device.init: call Swapchain.init(gpa, dev.handle, &phys_device, surface.handle, config.width, config.height)
  • RenderPass.init(dev.handle, swapchain.format)
  • Framebuffer.init(gpa, dev.handle, render_pass.handle, swapchain.views, swapchain.extent)
  • CommandPool.init(dev.handle, phys_device.queue_families.graphics)
  • Sync.init(dev.handle)

Extend deinit() — reverse order

  • vkDeviceWaitIdle(self.dev.handle) first — ensure GPU is done before tearing down
  • Sync.deinit(&self.sync, self.dev.handle)
  • self.cmd.deinit(self.dev.handle)
  • Framebuffer.deinit(self.gpa, self.dev.handle, self.framebuffers)
  • self.render_pass.deinit(self.dev.handle)
  • self.swapchain.deinit(self.dev.handle, self.gpa)
  • Existing: gpu.deinit, surface.deinit, instance.deinit

setClearColor()

  • Store as VkClearColorValue: self.clear_color = .{ .float32 = .{ color.r, color.g, color.b, color.a } };

beginFrame()

  • vkWaitForFences(self.dev.handle, 1, &s.in_flight, VK_TRUE, maxInt(u64))
  • vkAcquireNextImageKHR(self.dev.handle, self.swapchain.handle, maxInt(u64), s.image_available, null, &self.current_image)
  • Handle return value: VK_ERROR_OUT_OF_DATE_KHR → return error.SwapchainOutOfDate; VK_SUBOPTIMAL_KHR → continue (present will handle it)
  • vkResetFences(self.dev.handle, 1, &s.in_flight) — reset after acquire (not before, to avoid reset before check)
  • vkResetCommandBuffer(cb, 0)
  • VkCommandBufferBeginInfo: flags = VK_COMMAND_BUFFER_USAGE_ONE_TIME_SUBMIT_BIT
  • vkBeginCommandBuffer(cb, &begin_info)
  • VkRenderPassBeginInfo:
    • renderPass = self.render_pass.handle
    • framebuffer = self.framebuffers[self.current_image]
    • renderArea = { .offset = {0,0}, .extent = self.swapchain.extent }
    • clearValueCount = 1, pClearValues = &VkClearValue{ .color = self.clear_color }
  • vkCmdBeginRenderPass(cb, &rp_info, VK_SUBPASS_CONTENTS_INLINE)

endFrame()

  • vkCmdEndRenderPass(cb)
  • vkEndCommandBuffer(cb)
  • VkSubmitInfo:
    • waitSemaphoreCount = 1, pWaitSemaphores = &s.image_available
    • pWaitDstStageMask = &VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT
    • commandBufferCount = 1, pCommandBuffers = &cb
    • signalSemaphoreCount = 1, pSignalSemaphores = &s.render_finished
  • vkQueueSubmit(self.dev.graphics_queue, 1, &submit_info, s.in_flight)
  • VkPresentInfoKHR:
    • waitSemaphoreCount = 1, pWaitSemaphores = &s.render_finished
    • swapchainCount = 1, pSwapchains = &self.swapchain.handle
    • pImageIndices = &self.current_image
  • vkQueuePresentKHR(self.dev.present_queue, &present_info)
  • Handle VK_ERROR_OUT_OF_DATE_KHR or VK_SUBOPTIMAL_KHR → call recreateSwapchain()
  • self.current_frame = (self.current_frame + 1) % frames_in_flight

Update types.zig

  • Add re-exports: pub const Swapchain = @import("support/Swapchain.zig"); etc. for all five new modules
Reset fence AFTER acquire, not before. If you reset the fence before vkAcquireNextImageKHR and acquire returns an error (OUT_OF_DATE), you never re-signal the fence — the next frame waits forever. Acquire first, then reset only on success.
Phase 8

Resize — Swapchain Recreation modify

When the window is resized, the swapchain extent is stale. Vulkan signals this via VK_ERROR_OUT_OF_DATE_KHR on acquire or present. The response: wait for the GPU to finish, tear down the swapchain-dependent objects, and rebuild.

recreateSwapchain(width, height) — internal helper

  • vkDeviceWaitIdle(self.dev.handle)
  • Framebuffer.deinit(self.gpa, self.dev.handle, self.framebuffers)
  • self.swapchain.deinit(self.dev.handle, self.gpa)
  • self.swapchain = try Swapchain.init(..., width, height)
  • self.framebuffers = try Framebuffer.init(gpa, dev, render_pass, swapchain.views, swapchain.extent)
  • RenderPass can be reused — it does not depend on swapchain extent or image count
  • CommandPool and sync objects can be reused — they don't reference swapchain images

resize() — public API

  • pub fn resize(self: *Self, width: u32, height: u32) !void
  • Call recreateSwapchain(width, height)
  • renderer.zig already has a resize stub that delegates to the backend — wire it through
Not all objects need to be recreated on resize. The render pass, command pool, and sync primitives are extent-independent and can survive swapchain recreation. Only the swapchain, image views (inside Swapchain), and framebuffers are tied to specific image dimensions.

Verification

In order:

  • zig build zixelart compiles cleanly — no Vulkan validation errors at startup
  • Window appears on screen filled with the default clear color (black)
  • setClearColor changes the window color — visible on next frame
  • Resize the window — no crash, colors still display correctly
  • Close the window — clean shutdown, no GPU-side errors

Out of Scope (for now)

  • Graphics pipeline, shaders, vertex buffers — no draw calls in this phase
  • Depth/stencil buffer — color-only for now
  • Multiple render passes or subpasses
  • Validation layer setup — useful but separate from the frame machinery
  • MSAA (multi-sample anti-aliasing)
  • Multiple windows or surfaces
  • Split graphics/present queue families — Device.zig already rejects this; not needed on modern desktop GPUs
What comes after: Once beginFrame/endFrame work, all future rendering (geometry pipeline, texture sampling, UI) inserts draw calls between them without touching this infrastructure. The swapchain, render pass, and sync primitives become a stable foundation.
